Experiment 1: Partner-Identity-Conditioned Behavioral Adaptation in a Large Language Model

Methods — Draft for peer review
Model: LLaMA-2-13B-Chat  |  10 labeling versions

1. Overview

This experiment tests whether a large language model (LLM) adjusts its conversational behavior depending on whether it believes its conversation partner is a human or an AI. Fifty independent instances of LLaMA-2-13B-Chat each completed 40 simulated conversations—half with a partner labeled as human, half with a partner labeled as AI—on a fixed set of topics. Each conversation was analyzed across 23 linguistic measures spanning hedging, discourse markers, disfluency, theory-of-mind language, politeness, sentiment, and self-rated interaction quality. To isolate the source of any behavioral effects, the experiment was run under 10 different labeling conditions that systematically varied how partner identity was communicated in the system prompt: from specific partner names, to generic type labels, to nonsense control framings in which the identity tokens were present but semantically decoupled from the conversational context.

2. Model and Generation Parameters

All conversations were generated using LLaMA-2-13B-Chat (meta-llama/Llama-2-13b-chat-hf), a 13-billion-parameter instruction-tuned language model fine-tuned with RLHF (Touvron et al., 2023). The model was loaded in half-precision (float16) via HuggingFace Transformers with device_map="auto" for automatic GPU placement. Generation used nucleus sampling with the following parameters: temperature = 0.8, top-p = 0.95, maximum new tokens = 500. Pad token ID was set to the EOS token ID. A single model instance served both conversational roles (participant and partner), differentiated only by their system prompts and conversation histories.

3. Experimental Design

3.1 Participants

Fifty independent model instances (hereafter “subjects,” indexed s001–s050) were run, each as a separate SLURM job on a GPU node (1 GPU, 32 GB RAM, 8-hour time limit). Each subject received a unique pseudorandom condition assignment file specifying the order of partners and topics across 40 trials.

3.2 Conditions

Each subject completed 40 conversations organized into 5 runs of 8 trials. Within each run, four agent conditions appeared: bot_1, bot_2 (AI partner), hum_1, and hum_2 (human partner). Each agent condition was paired with one social and one nonsocial topic per run, yielding 8 trials per run. Across all 5 runs, each subject completed 20 human-partner conversations and 20 AI-partner conversations, balanced across 40 unique topics. The mapping from agent keys to partner labels varied across versions (see Section 4).

3.3 Topics

Forty conversational topics were divided equally into 20 social and 20 nonsocial categories. Each topic was operationalized with a 1–2 sentence prompt (e.g., “Trust: Discuss what makes it easy or difficult to trust someone.”). Topics were counterbalanced across conditions.

Social topics (20)Nonsocial topics (20)
hobbies, friendship, childhood memories, conflict, empathy, mentorship, role models, networks, love, loneliness, communication, trust, teamwork, work–life balance, forgiveness, boundaries, family, goals, social media, cultural celebrations nature, cars, space, photography, books, art, exercise, sports, music, architecture, cooking, movies, food, technology, travel, seasons, fashion, weather, podcasts, gardening

3.4 Conversation Structure

Each conversation consisted of 5 exchange turns between the participant model and a partner model. The participant spoke first. On each subsequent turn, the partner’s reply was appended to the participant’s conversation history as a user message (with a version-dependent prefix; see Section 4.3), and the participant generated its next response. After the fifth exchange, the participant was prompted to rate the conversation on two 4-point scales (quality and connectedness). Ratings were parsed from JSON output and clipped to [1, 4], with a default of (2, 2) if parsing failed.

Conversation history was maintained using the LLaMA-2-Chat template format:

<s>[INST] <<SYS>> {system_prompt} <</SYS>> {user_msg_1} [/INST] {asst_msg_1} </s><s>[INST] {user_msg_2} [/INST] ...

All system messages were retained throughout the conversation. The most recent 5 user–assistant pairs (i.e., the full 5-turn history) were included in context; no truncation occurred at this conversation length.

3.5 Role Prompts

The participant received a system prompt that included the identity manipulation (see Section 4), plus a topic introduction as its first user message:

The conversation topic is: '{topic_text}'. Please begin by producing only your first message to start the conversation. Do not simulate both sides of the dialogue.

The partner received a generic system prompt with no identity information:

You are engaging in a real-time spoken conversation. Begin each response with your actual message content.

Both agents received the same topic introduction. The partner had no knowledge of whether it was supposed to be “human” or “AI”—only the participant’s system prompt carried the identity manipulation.

3.6 Post-Conversation Ratings

After the fifth exchange, the participant rated the conversation:

You are rating this conversation you just had. 1 = very poor/disconnected, 2 = somewhat poor, 3 = good, 4 = excellent. Rate: 1) Overall conversation quality (1-4) 2) How connected you felt to your partner (1-4) Be realistic -- vary ratings depending on how natural or awkward the exchange felt. Return ONLY a JSON object like: {"quality": <1-4>, "connectedness": <1-4>} No explanation, no extra keys.

4. Labeling Versions

The experiment was run 10 times with systematically varied labeling strategies. All versions shared identical generation parameters, conversation structure, topics, and analysis pipeline. The only variation was the participant’s system prompt template and, in some versions, the prefix prepended to partner messages in the conversation history.

4.1 Participant System Prompt Templates

All templates shared a common frame:

You are the participant in a brief conversation. [IDENTITY CLAUSE]. Stay in character as the participant throughout. Speak naturally and share your own thoughts and experiences.

The identity clause varied by version:

4.3 Turn Prefix Format

When partner messages were added to the participant’s conversation history, they were prefixed with a label. The prefix format varied by version and constituted an additional channel of identity reinforcement (or suppression):

Version groupPrefix formatExample
Named-partner versions{name}: {message}ChatGPT: Hello there!
Turnwise versionsHuman: {message} / AI: {message}AI: Hello there!
Labels, nonsense, you_are_labelsPartner: {message}Partner: Hello there!

Design rationale. Named-partner versions were designed to maximize ecological validity but introduced confounds: the model may have adapted to stored associations with “ChatGPT” or “Sam” rather than to the abstract identity category. The labels version removed all name-based cues, leaving only the type token (“a Human” / “an AI”) in the system prompt and a neutral “Partner:” turn prefix. The nonsense controls preserved the same identity tokens but placed them in non-instructional contexts (codeword assignment or ignore instruction), testing whether the model’s behavioral adaptation requires semantic understanding of the identity clause or merely the presence of the tokens.

5. Linguistic Measures

Twenty-three measures were computed for each conversation, spanning six categories. All lexical measures used case-insensitive regex matching with word-boundary anchors (\b) applied to participant utterances only (not partner speech).

5.1 Hedging (Demir, 2018)

Following the six-category taxonomy of Demir (2018; based on Hyland, 1998), hedging markers were counted in six subcategories plus a total:

CategoryItemsExamples
Modal auxiliaries6can, could, may, might, should, would
Epistemic verbs44 lemmas (132 inflected forms)believe, suggest, seem, think, assume, suppose, indicate, speculate, ...
Epistemic adjectives19likely, possible, probable, plausible, uncertain, potential, approximate, ...
Epistemic adverbs47perhaps, probably, possibly, roughly, approximately, somewhat, fairly, ...
Quantifiers14a few, most, several, more or less, to some extent, ...
Epistemic nouns19possibility, assumption, belief, estimate, suggestion, tendency, ...
TotalSum of all six subcategory counts

Each subcategory rate was computed as count / word count. The total rate was computed from summed counts (not averaged subcategory rates).

5.2 Discourse Markers (Fung & Carter, 2007)

Twenty-three discourse markers from Fung and Carter’s (2007) frequency analysis were counted in four functional subcategories:

CategoryItemsMarkers
Interpersonal26 patternsyou know, you see, well, really, I think, obviously, basically, actually, sort of, kind of, like, just, oh, okay, right, yeah, yes, sure, great, ...
Referential14 patternsbecause, cos, but, yet, however, nevertheless, and, or, so, anyway, likewise, similarly
Structural25 patternsnow, okay, right, well, let’s start, first, secondly, then, finally, so, how about, what about, and, ...
Cognitive12 patternswell, I think, I see, I mean, that is, in other words, like, sort of, kind of, you know

Some markers appear in multiple subcategories (e.g., “well” in interpersonal, structural, and cognitive). Subcategory rates are therefore not mutually exclusive. The total discourse marker rate uses the combined list of 23 unique markers from Fung and Carter’s Table 2 (24 regex patterns, as “okay” and “ok” are matched separately).

5.3 Disfluency (LIWC, 2007)

Two categories from the Linguistic Inquiry and Word Count dictionary (Pennebaker, Booth, & Francis, 2007) plus a combined total:

5.4 Discourse “Like”

Occurrences of discourse “like” (encompassing approximator, quotative, and filler uses; D’Arcy, 2007) were counted using the pattern \blike\b. This captures all uses of “like” as a standalone word, including non-discourse uses (e.g., “I like pizza”). The rate was computed as count / word count.

5.5 Theory-of-Mind Phrases

Second-person mental state attributions were counted using 38 regex patterns of the form “you + cognitive verb” (following Wagovich et al., 2024): you think, you know, you believe, you understand, you remember, you realize, you imagine, you wonder, you expect, you consider, you decide, etc. The rate was computed as count / word count.

5.6 Politeness

A composite politeness score was computed from three word lists:

CategoryItemsExamples
Positive politeness4 patternsthank(s/you/ful), appreciate, great/wonderful/awesome/excellent, hello/hi/hey
Negative politeness8 patternssorry, please, could you, would you, might you, if you could, perhaps, by any chance
Impoliteness4 patternsyou need to, you should, do not, in fact

The politeness score was: (positive_count + negative_count − impolite_count) / word_count.

5.7 Sentiment

Sentiment was measured using VADER (Valence Aware Dictionary and sEntiment Reasoner; Hutto & Gilbert, 2014), applied via nltk.sentiment.SentimentIntensityAnalyzer. The compound score (range −1 to +1) was used, representing overall sentiment valence. Unlike count-based measures, sentiment was averaged (not summed) across turns within a trial.

5.8 Self-Reported Ratings

Two 4-point ratings were collected per conversation: overall conversation quality and connectedness to the partner. These were trial-level measures (one value per conversation), averaged across trials within each condition.

6. Aggregation

Linguistic features were aggregated in three steps:

Step 1: Sum within trial. For each conversation (subject × condition × topic), raw counts of each marker category were summed across all 5 turns. Word count was also summed. Sentiment, quality, and connectedness were averaged across turns (or taken at the trial level for ratings).

Step 2: Recompute rates. At the trial level, each rate metric was recomputed as: rate = summed_count / summed_word_count. Trials with zero total word count were excluded (set to NaN). This approach weights each turn by its word count, preventing short turns from disproportionately influencing rates.

Step 3: Average to subject × condition. Trial-level rates were averaged across all 20 trials within each condition (Human or AI) for each subject, yielding one Human mean and one AI mean per subject per metric. These subject-level means served as the input to statistical tests.

Why sum-then-rate rather than average rates? Summing counts before computing rates avoids inflating the contribution of short utterances. A 5-word turn with 1 hedge (rate = 0.20) would dominate a 100-word turn with 3 hedges (rate = 0.03) under rate-averaging, but contributes proportionally under the sum-then-rate approach. This is the standard procedure in corpus linguistics (see Biber, 1988).

7. Statistical Analysis

7.1 Primary Test: Paired t-Test

For each of the 23 metrics, a two-tailed paired-samples t-test compared the Human-condition mean against the AI-condition mean across 50 subjects (df = 49). The test was implemented via scipy.stats.ttest_rel with nan_policy="omit".

7.2 Effect Size

Effect size was quantified as Cohen’s dz for paired designs:

dz = Mdiff / SDdiff

where Mdiff is the mean of (Human − AI) difference scores and SDdiff is the standard deviation of the differences (with Bessel’s correction, ddof = 1).

7.3 Multiple Comparison Correction

The Benjamini–Hochberg (BH) procedure was applied across all 23 metrics to control the false discovery rate at α = 0.05. The implementation ranked p-values in ascending order, computed adjusted p-values as padj = min(1.0, p × m / rank) where m = 23, then enforced monotonicity via a backward pass: padj[i] = min(padj[i], padj[i+1]). Significance thresholds were: ★ padj < .05, ★★ padj < .01, ★★★ padj < .001.

7.4 Secondary: Condition × Topic Sociality ANOVA

For named-partner versions (which support a 2 × 2 within-subject design), a repeated-measures ANOVA tested the interaction of Condition (Human vs. AI) × Topic Sociality (social vs. nonsocial). The ANOVA was computed using manual sums-of-squares decomposition with F-statistics and p-values derived from the F-distribution (scipy.stats.f.cdf). Degrees of freedom were (1, n − 1) for each main effect and the interaction. When the interaction was significant (p < .05), simple effects were tested via paired t-tests within each level of Topic Sociality.

For the labels version (which collapses individual partner identities into two type-level conditions), the analysis used only the main paired t-test, as there was no individual identity variation to support within-type comparisons.

8. Cross-Version Analysis

8.1 Behavioral Summary

For each of the 10 versions, the same 23-metric analysis pipeline was applied. Results were aggregated into a summary table displaying the t-statistic, p-value, effect direction (Human > AI or AI > Human), and BH-corrected significance for each metric × version combination.

8.2 Per-Turn Analysis

To examine how effects evolved across the conversation, each metric was analyzed separately at each of the 5 turns. Within each turn, a paired t-test (Human vs. AI) was computed across subjects, with BH-FDR correction applied across the 21 per-utterance metrics within each turn. This yielded a turn × version × metric array of significance indicators, revealing whether behavioral differences emerged immediately or accumulated over the conversation.

8.3 Per-Identity Analysis

For named-partner versions, within-type comparisons (e.g., ChatGPT vs. Copilot, Sam vs. Casey) and cross-type comparisons (Human vs. AI) were computed with paired t-tests and BH-FDR correction, diagnosing whether effects were driven by specific partner names or by the abstract type category.

10. Design Considerations

10.1 Same Model for Both Roles

A single LLaMA-2-13B-Chat model instance served both conversational roles. The participant and partner were differentiated solely by their system prompts and conversation histories. This ensured that any behavioral differences were attributable to the system prompt manipulation, not to architectural differences between agents. A potential concern is cross-contamination of internal states; however, the model processes each role’s history independently via the standard transformer forward pass, and the two roles never share a conversation history.

10.2 Temperature Selection

Temperature = 0.8 was chosen to balance response diversity with coherence. Lower temperatures would produce more deterministic (and potentially more uniform) responses, reducing statistical power to detect condition effects. Higher temperatures would increase variability but risk incoherent generations.

Flag: Lexical overlap between measures. Several markers appear in multiple measurement categories (e.g., “well” in both LIWC nonfluencies and Fung discourse markers; “you know” in both LIWC fillers and Fung interpersonal markers). Subcategory rates are therefore not fully independent. The BH-FDR correction partially addresses this by controlling the discovery rate across all tests, but correlated measures inflate the effective number of tests relative to the nominal count of 23.
Flag: Discourse “like” captures all uses. The regex \blike\b matches all occurrences of “like” including non-discourse uses (e.g., “I like pizza”). An elevated rate in one condition could reflect increased discourse “like” (filler, approximator, quotative) or increased expression of preferences. Manual inspection of a subsample is advisable to characterize the source of any significant difference.
